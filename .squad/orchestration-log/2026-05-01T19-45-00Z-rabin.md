### Critical Application Responsibility Shift
**Current (backend injection):** `get_user_household_id(db, user_id)` resolves household  
**Future (frontend-direct):** Frontend reads household_id from Supabase Auth JWT; passes in all CRUD requests  

**No database auto-injection:** No triggers, no `current_setting()`, no column DEFAULT (intentional). Frontend is single source of truth for household scoping.

### Top 3 Risks & Mitigations
1. **Malicious household_id:** RLS rejects (enforced at DB). Mitigation: Frontend must NOT expose household_id as user input; source from session JWT/profile.
2. **Frontend omits household_id:** RLS `NOT NULL` check rejects. Mitigation: TypeScript types mark household_id as required, not optional.
3. **Viewer escalates to writer:** RLS uses `is_household_writer()` = (role IN ('owner', 'member')). Mitigation: Frontend respects viewer role; DB enforces at RLS layer.
