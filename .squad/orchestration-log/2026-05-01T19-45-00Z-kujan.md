## Key Outputs

### Python Backend Hosting Decision
**Question:** Can FastAPI run on Vercel Functions?  
**Answer:** No. Keep local Docker.

**Vercel Constraints:**
- 60s max execution (backtests exceed this)
- Ephemeral filesystem (breaks IB Gateway sockets)
- No native WebSocket/long-poll support
- Cold starts 8–15s (blocks interactive requests)

**Backend Workloads:**
- `POST /api/backtest/run` — compute-heavy, pandas/scipy/numpy
- `GET /api/trading/*` — IB Gateway persistent sockets
- Scheduled imports (IBKR/Schwab token sync)
- Background workers (async)

**Current Architecture Approved:**
- Frontend: Vercel (Next.js)
- Backend: Local Docker (dev) → Render/Railway/Fly.io (prod)
- Database: Supabase
- No changes required; topology sound

### API Rewrite Hardening
**Problem:** Production deployments silently fail when `NEXT_PUBLIC_API_URL` is misconfigured or missing  
**Solution:** next.config.ts validation:
- Local dev: fallback to `http://127.0.0.1:8000`
- Production: fail loudly on missing, empty, malformed, localhost, loopback, private-address values
- Non-HTTP(S) URLs rejected
- Result: Early detection, no silent prod failures

**Open Decision:** Backend deployment strategy — user must choose:
1. Deploy FastAPI publicly, set `NEXT_PUBLIC_API_URL` to public URL
2. Port API endpoints to Next.js route handlers (Vercel-owned)

Production write paths remain broken until decision made and implemented.
