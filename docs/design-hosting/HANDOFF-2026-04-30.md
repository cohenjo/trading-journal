### Step 3 — Push env vars to Vercel (CLI, after Step 1+2)

> **Two-environment model:** We only have 2 Supabase free-tier projects (dev + prod), so we skip Vercel's `preview` environment entirely. Preview branch builds will fall back to `development` env vars (which point at the `trading-journal-dev` Supabase project).

Run from `apps/frontend/` once `vercel link` is done. Each command prompts for the value (no echo):

```bash
# ─── Production (points at trading-journal-prod Supabase) ──────────────
vercel env add NEXT_PUBLIC_SUPABASE_URL        production    # https://<PROD_PROJID>.supabase.co
vercel env add NEXT_PUBLIC_SUPABASE_ANON_KEY   production    # prod anon key
vercel env add NEXT_PUBLIC_SITE_URL            production    # https://<your-domain> or trading-journal.vercel.app
vercel env add SUPABASE_SERVICE_ROLE_KEY       production    # prod service_role 🔴
vercel env add SUPABASE_JWT_SECRET             production    # only if HS256 fallback needed

# ─── Development (points at trading-journal-dev Supabase) ──────────────
# Used for `vercel env pull` locally AND for any preview branch build
# (since we deliberately don't have a preview env)
vercel env add NEXT_PUBLIC_SUPABASE_URL        development   # https://<DEV_PROJID>.supabase.co
vercel env add NEXT_PUBLIC_SUPABASE_ANON_KEY   development   # dev anon key
vercel env add NEXT_PUBLIC_SITE_URL            development   # http://localhost:3000
vercel env add SUPABASE_SERVICE_ROLE_KEY       development   # dev service_role 🔴
vercel env add SUPABASE_JWT_SECRET             development   # optional (HS256 fallback)

# Verify
vercel env ls production
vercel env ls development

# Pull development env locally (does NOT clobber if values match)
vercel env pull apps/frontend/.env.local
```

> **Note about Vercel preview deploys:** PR/branch deploys on Vercel default to the `preview` env. Without `preview` vars set, Vercel will fall back to `development` values. That's intentional for our setup — preview builds talk to `trading-journal-dev`, never prod. If you ever upgrade to a 3rd Supabase project, you can split out `preview` env vars at that time.

### Step 4 — Configure Supabase Auth (dashboard or PAT)

For **both** dev and prod projects:

1. Dashboard → **Authentication → URL Configuration**:
   - **Site URL** (dev): `http://localhost:3000`
   - **Site URL** (prod): your real domain or `https://trading-journal.vercel.app`
   - **Redirect URLs** (add all): 
     ```
     http://localhost:3000/auth/callback
     https://trading-journal-*-cohenjo.vercel.app/auth/callback
     https://<your-prod-domain>/auth/callback
     ```
   > Branch/preview deploys also land on `trading-journal-*-cohenjo.vercel.app`. Both dev-remote testing and prod sign-in go through this same allowlist, so add the wildcard once per project.

2. Dashboard → **Authentication → Providers → Google**:
   - Enable Google.
   - Set **Client ID** + **Client Secret** from <https://console.cloud.google.com/apis/credentials>.
   - **Authorized redirect URIs in Google Cloud:** `https://<projid>.supabase.co/auth/v1/callback` (one per project).
   - **Authorized JavaScript origins in Google Cloud:** your Vercel domain(s) + `http://localhost:3000`.

(See `docs/design-hosting/runbooks/supabase-03-auth-rls.md` if it exists, or `docs/design-hosting/sections/03-auth-sharing-security.md` for the full flow.)

```bash
# DEV — schema operations
PGPASSWORD="$SUPABASE_DEV_PASS" psql \
  "host=aws-1-eu-central-1.pooler.supabase.com port=5432 \

# PROD — same shape, swap PASS + PROJID
```

> **Not a typo:** the pooler hostname is `aws-1-…` not `aws-0-…`. Older Supabase docs say `aws-0-…`; that's the legacy fleet. Both projects above are on the new `aws-1-…` fleet.
